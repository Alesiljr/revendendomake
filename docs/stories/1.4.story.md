## Description

Este story aplica o DDL completo no banco de dados Supabase (tabelas, enums, índices, triggers e RLS policies), cria os buckets de Storage com restrições de MIME type e tamanho, configura a conta de admin via Supabase Auth, insere os dados iniciais via seed.sql, e gera os tipos TypeScript do schema.

Todo o DDL de referência está em `docs/fullstack-architecture.md` seções 3.1 e 3.2 — copie diretamente de lá para `supabase/migrations/001_initial_schema.sql`. Este story não cria novos dados de negócio (produtos, depoimentos, artigos) — apenas a estrutura e uma row em `site_settings`.

Ao final, o banco estará 100% operacional, as RLS policies testadas e aprovadas, os tipos TypeScript gerados e commitados, e a conta admin pronta para uso. Este é o pré-requisito técnico mais crítico para todos os stories dos Epics 2, 3 e 4.

### Triggers updated_at

- [ ] AC10: Função trigger criada e aplicada a todas as tabelas que têm `updated_at`:
  ```sql

    RETURN NEW;
  END;
  $$ LANGUAGE plpgsql;
  ```

### RLS — Pontos de Atenção

- A policy de `products` usa `auth.role() = 'authenticated'` para permitir que admins vejam produtos inativos. Certifique-se de que esta função está disponível no Supabase (é nativa do `@supabase/auth-helpers`).
- Nunca usar `service_role` key no cliente browser (exporia dados sem RLS). O `SUPABASE_SERVICE_ROLE_KEY` é apenas para Server Actions e API Routes server-side.
- O `anon` key é seguro para expor no frontend pois o RLS protege os dados.

### Tipos TypeScript — Uso Correto

Após gerar `src/lib/supabase/types.ts`, use os tipos assim:

```typescript
import type { Database } from '@/lib/supabase/types'

// Tipo derivado da tabela (Row = todos os campos)
type Product = Database['public']['Tables']['products']['Row']

// Tipo para INSERT (sem id, created_at, updated_at)
type ProductInsert = Database['public']['Tables']['products']['Insert']

// Tipo para UPDATE (todos opcionais)
type ProductUpdate = Database['public']['Tables']['products']['Update']
```

Nunca criar tipos manualmente para o schema — sempre derivar dos tipos gerados.

### site_settings — Singleton Pattern

A tabela `site_settings` tem constraint `CHECK (id = 1)` — só pode existir uma row. Use sempre `upsert` ou `update` via admin, nunca `insert`. A aplicação deve assumir que esta row sempre existe (o seed.sql garante isso).

- Supabase Realtime — não necessário para v1
